# 管理后台周期计划列表重构设计

日期: 2026-03-27
范围: 管理后台周期管理、周期自动推进、未来周期生成策略、相关前后端接口

## 背景

当前系统把“调度配置”“周期生成”“状态推进”“管理员手动操作”拆在几套机制里:

- 调度配置决定一组开关时间
- 到点后系统再生成新周期
- 读取接口和手动同步仍承担一部分状态推进职责
- 管理员看到的是当前状态，而不是完整计划

这导致三个可用性问题:

1. 管理员无法直接看到未来几周的完整安排。
2. 某一周如果需要微调时间，必须先理解调度配置和自动生成逻辑。
3. 后台同时存在“周期列表”和“调度器配置”，心智模型重复。

目标是把“周期列表”升级成唯一的计划真源，让管理员直接管理未来周期，而不是管理一套抽象调度规则。

## 目标

1. 默认预生成未来 20 个周期。
2. 每个周期都可在列表中直接查看开始/结束时间。
3. 管理员可以在列表中直接调整未来周期时间，必要时删除未来周期。
4. 到达周期时间后，系统只按“周期列表中已存在的时间”推进状态，不再在到点时临时生成新周期。
5. 删除后台“自动调度配置”区块，避免和周期列表重复。
6. 删除后台“结果预览”区块，结果统一通过结果公示专页查看。

## 非目标

1. 不改变成员评分、组长评分、结算公式。
2. 不改变“结算”和“公示”是两个显式动作这一重构方向。
3. 不在本轮引入复杂的批量编辑、拖拽排期或甘特图。

## 方案概览

推荐方案: `周期计划列表作为唯一时间源`

核心思路:

- 数据库中始终维护一段连续的未来周期窗口，默认窗口长度为 20 周。
- 每个周期都拥有自己的 `start_at`、`end_at`，管理员可单独调整。
- 系统启动或管理员进入后台时，只负责“补齐窗口”，不再依赖独立调度配置表决定未来周期开关时间。
- 定时任务只做“扫描现有周期并推进状态”，不再负责“生成新周期”。

这样管理员只需要理解一件事:

> 周期什么时候开始、什么时候结束，以周期列表里那一行的时间为准。

## 数据模型

继续沿用 `rating_cycles`，不新增新的计划表。

新增/约定字段语义:

- `week_number`: 连续周编号，作为周期顺序主键语义。
- `start_at` / `end_at`: 周期的唯一时间来源。
- `status`: `draft | active | closed | settled`
- `settle_mode`: 记录结算来源，不再承担排期配置职责。

不再依赖:

- `scheduling_config` 作为未来周期时间模板的真源。

保留最小兼容策略:

- 数据库表可暂时保留，避免迁移时直接删表。
- 前端和业务逻辑不再读取或展示该配置。

## 后端设计

### 1. 周期窗口补齐

新增一个“计划窗口补齐”能力:

- 若当前最大 `week_number` 之后的未来周期少于 20 周，则自动补齐到 20 周。
- 新补齐周期默认按“上一周起始时间 + 7 天”计算。
- 新补齐周期默认持续时间为 2 天。

补齐触发点:

- `db:init`
- 应用启动
- 管理后台获取周期列表时
- 删除未来周期后

### 2. 周期状态推进

保留一个很薄的定时任务或启动时同步逻辑，但职责收缩为:

- `start_at <= now < end_at` 的 `draft` 周期推进为 `active`
- `end_at <= now` 的 `draft|active` 周期推进为 `closed`

不再负责:

- 新建周期
- 解释“周几几点”的抽象调度配置

### 3. 周期列表接口

管理员周期列表接口返回:

- 当前窗口内全部周期
- 每个周期的阶段标签
- 每个周期可执行动作

动作规则:

- `draft`: 可调整时间、可删除
- `active`: 可提前结算
- `closed`: 可结算
- `settled` 且未公示: 可公示
- `published`: 可归档

### 4. 默认时间策略

默认计划锚点:

- 以当前工作周期为基准
- 后续周期统一使用“上一周期开始时间 + 7 天”
- 周期持续时间默认 2 天

如果管理员改了某一周:

- 仅修改该周，不自动级联改后续 19 周
- 后续新补齐周期继续以“最后一条已有周期”为锚点往后排

这样行为最可预期，也最不容易误伤管理员手工改过的时间。

## 前端设计

### 1. 移除区块

从管理员后台移除:

- 自动调度配置区
- 结果预览区
- “当前公示周期”摘要卡

### 2. 周期控制中心

保留:

- 当前工作周期
- 待公示周期
- 一个主动作按钮

移除:

- 当前公示周期摘要卡

这样控制中心只告诉管理员:

- 现在正在做哪一周
- 有没有一周已经结算完但还没公示
- 下一步该点什么

### 3. 周期列表

周期列表改成后台主操作区，单行显示:

- 周期名称
- 时间范围
- 阶段标签
- 操作按钮

按钮规范统一:

- 同高
- 同水平内边距
- 删除按钮仅颜色不同，不再更小

每行操作:

- `draft`: 调整时间、删除
- `active`: 提前结算
- `closed`: 立即结算
- `settled`: 公示
- `published`: 归档

### 4. 默认 20 周体验

管理员第一次打开后台就能看到:

- 已完成周
- 当前周
- 后续 20 周计划

这会明显降低“系统下一步会不会自己生成奇怪周期”的不确定性。

## 迁移策略

### 阶段 1

- 先隐藏调度配置 UI
- 后端保留 `scheduling_config` 表，但前端不再使用
- 定时任务改为只推进状态，不再生成周期

### 阶段 2

- 新增“补齐 20 周窗口”逻辑
- 周期列表接口返回完整窗口

### 阶段 3

- 后台周期列表支持未来周期调整/删除/补齐后刷新
- 控制中心和列表动作收敛到同一套规则

### 阶段 4

- 清理旧调度文案、旧接口引用
- 如确认不再需要，再考虑删除 `scheduling_config` 表及相关代码

## 风险与对策

### 风险 1: 现有数据中未来周期不连续

对策:

- 补齐逻辑只依赖当前最大 `week_number` 和最后一个周期的开始时间
- 对缺口按顺序补，不尝试“智能修复”管理员历史改动

### 风险 2: 管理员误删未来周期导致计划窗口变短

对策:

- 删除后自动补齐到 20 周
- 仅允许删除 `draft` 周期

### 风险 3: 改动某周时间后与下一周重叠

对策:

- 保存时校验时间范围
- 禁止与相邻周期重叠
- 错误提示明确指出冲突周次

## 测试策略

后端:

- 初始化后自动补齐到 20 周
- 删除未来周期后自动补齐
- 修改未来周期时间不会改变其他周期
- 到点后只推进既有周期状态，不生成新周期

前端:

- 后台不再显示调度配置区
- 后台不再显示结果预览区
- 周期列表按钮尺寸一致
- `draft` 周期显示“调整时间 / 删除”
- `active` 周期显示“提前结算”
- `settled` 周期显示“公示”

E2E:

- 管理员进入后台可看到连续计划周
- 调整第 4 周时间后刷新仍保留
- 删除第 6 周后系统补齐一个新的第 20+ 周
- 到达当前周结束时间后自动进入待结算状态

## 推荐实施顺序

1. 先做后台 UI 收敛: 去掉重复区块、统一按钮。
2. 再做后端 20 周窗口补齐。
3. 然后把周期列表动作补全。
4. 最后移除旧调度配置依赖和相关测试替换。
